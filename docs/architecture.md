## 数据采集流程

```
用户点击"刷新" 或 定时器触发
        │
        ▼
  检查网络状态
        │
        ▼
  遍历 5 个区域（禅城→南海→顺德→三水→高明）
        │
        ▼
  请求项目列表 → 增量更新 projects 表
        │
        ▼
  对每个项目：请求楼栋列表 → 增量更新 buildings 表
        │
        ▼
  对每个楼栋：请求房源列表 → 增量更新 units 表
        │
        ▼
  对每个房源：提取面积、总价 → 计算单价、9折后单价
        │
        ▼
  写入 price_snapshots（当天日期，同房源覆盖）
        │
        ▼
  标记 sync_log（待同步）
        │
        ▼
  通知 UI 更新完成
        │
        ▼
  触发云端同步
```

## 数据关系

```
districts (1) ──< (N) projects (1) ──< (N) buildings (1) ──< (N) units (1) ──< (N) price_snapshots

一个区域 → 多个楼盘
一个楼盘 → 多个楼栋
一个楼栋 → 多个房源
一个房源 → 多条价格快照（时间序列）
```

## 数据库表结构

### districts（区域表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 英文标识，如 'chancheng' |
| name | TEXT | 中文名，如 '禅城区' |
| sort_order | INTEGER | 排序顺序 |

### projects（项目表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 网站上的项目唯一ID |
| district_id | TEXT FK | 所属区域 |
| name | TEXT | 项目名称 |
| address | TEXT | 详细地址 |
| developer | TEXT | 开发商 |

### buildings（楼栋表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 网站上的楼栋唯一ID |
| project_id | TEXT FK | 所属项目 |
| name | TEXT | 楼栋名称（如 '1栋'） |

### units（房源表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 网站上的房源唯一ID |
| building_id | TEXT FK | 所属楼栋 |
| project_id | TEXT FK | 所属项目 |
| room_number | TEXT | 房号（如 '101'） |
| area | REAL | 建筑面积（㎡） |
| status | TEXT | 状态（可售/已预定/已签约/已备案） |

### price_snapshots（价格快照表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | UUID |
| unit_id | TEXT FK | 关联房源 |
| total_price | REAL | 备案总价（元） |
| unit_price | REAL | 备案单价（元/㎡） |
| discounted_unit_price | REAL | 9折后单价（元/㎡） |
| snapshot_date | TEXT | 快照日期（YYYY-MM-DD） |
| scraped_at | TEXT | 采集时间戳 |


## 云端同步策略

- **上行同步**（本地 → 云端）：查询 sync_log 中 synced=0 的记录，逐条上传，标记 synced=1
- **下行同步**（云端 → 本地）：获取本地最后同步时间，拉取云端更新的数据，写入本地
- **冲突解决**：时间戳优先（Last-Write-Wins）
- **触发时机**：应用启动 + 采集完成 + 手动触发
